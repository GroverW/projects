Finally got a great explanation of this (or at least one that I can understand) via This Video, so it's time to try to cement this into my memory.

Inheritance

Inheritance is when you define your objects based on what they are:

Car class - cars have wheels and can drive, so in order to avoid duplicating the "wheels" and "drive" methods, any type of car will just inherit those from the car class.
Plane class - planes have wings and can fly, so any type of plane in teh plane class will have wings and be able to fly.

Ok that works pretty great up until the flying car is invented and then... we need to be able to fly, drive, have wheels and maybe even wings on our flying car.

To rectify the situation we could:

  • Make an even higher level class (vehicle?) that our planes and cars inherit from. But then when we create a new plane or car, we get a bunch of extra functionality we didn't ask for. I guess there's a popular analogy called the Gorilla-Banana problem where you ask for a banana and you get a gorilla holding the banana, and the entire jungle along with it.
  • Duplicate methods; put fly in the car class or something like that. This isn't as bad, but still isn't a great solution.
  • Or we could do...

Composition

Composition is when you define your objects based on what they do:

Now the relationship is reversed in a way:

Car class - a thing that drives and has wheels.
Plane class - a thing that flies and has wings.
Flying Car class - a thing that flies, drives, has wheels and maybe even wings.

Now drive, fly, wheels and wings are their own objects, and they can be incorporated into the car class or the boat class or any other class you can think up.


    const driver = (state) => ({
        drive: () => state.positionX += state.speed;
    })
    
    const flyer = (state) => ({
        fly: () => {
            state.positionX += state.lift;
            state.positionY += state.speed;
        }
    })
    
    const flyingCar = (name) => {
        let state = {
            name,
            speed: 100,
            lift: 10,
            positionX: 0,
            positionY: 0
        }
    
        return Object.assign{
            {},
            driver(state),
            flyer(state)
            //... the rest
        }
    }

Which is better? Is-A vs Has-A

Most programmers apparently say you should favor composition over inheritance. In the video, he argues for only using composition - for pretty much any class structure, you could look at it from both sides, and inheritance tends to be weaker because it requires you to predict the future about what types of objects you'll need to create. The example he gives is that he is-a man, therefore he can inherit from the man class, versus a car which has-an engine and could use composition. But, as a man he has arms and has cooking skills, therefore that could also be expressed using composition.

I like that reasoning as a person who knows so little at this point, but who knows how my opinion will change in the future.
